# 项目结构与调用链总览

> 目标：让刚接手项目的人，先用这一份文档快速知道「目录做什么」「页面连到哪里」「前端脚本和后端代码该从哪里看起」。

## 1. 先看结论

这个仓库当前是一个 **Go 学习 + 可运行实战示例** 的混合项目，真实可联调的主链路是：

```text
静态页面(web/*.html)
  → 页面内联脚本 / web/*.js
  → TODO API(cmd/todoapi + internal/todo)
  → 可选经过 Gateway(examples/gateway/gin 或 stdlib)
  → 存储层(内存 / SQLite / MySQL)
```

注意两点：

- 仓库里 **没有 Python 后端代码**，只有部分页面和文档还保留了旧的 Python/RBAC 演示口径。
- 当前最完整、最值得作为“主链路”阅读的是：
  - `web/todo-login.html`
  - `web/auth-helper.js`
  - `cmd/todoapi/main.go`
  - `internal/todo/handler.go`
  - `examples/gateway/gin/main.go`

---

## 2. 推荐阅读顺序

如果你是第一次接手，建议按这个顺序看：

1. `README.md`
   - 先理解整个仓库是“学习文档 + 示例代码 + 前后端演示”三合一结构。
2. `web/portal.html`
   - 看前端入口页都暴露了哪些学习区、项目区、联调入口。
3. `web/todo-login.html` + `web/auth-helper.js`
   - 这是最短的一条真实前后端链路。
4. `cmd/todoapi/main.go`
   - 看服务如何选择存储、如何初始化 `Server`。
5. `internal/todo/handler.go`
   - 看所有 HTTP 路由和业务入口。
6. `internal/todo/auth.go` / `security.go` / `user_store.go`
   - 看 JWT、refresh token、登录失败限制、用户查询。
7. `internal/todo/store.go` / `store_db.go`
   - 看数据是怎么落到内存或数据库里的。
8. `examples/gateway/gin/main.go`
   - 看网关如何把 `/api/v1/*` 转发到 TODO API。

## 3. 根目录主要结构

### 3.1 学习文档区

- `01.Go语言基础/`
  - Go 基础章节文档，偏语言入门。
- `02.开发环境及框架介绍/`
  - 工具链、框架、数据库、配置、日志等扩展内容。
- `03.项目实战/`
  - 项目实战说明文档。
- `docs/`
  - 面向当前仓库的说明文档、链路文档、接口文档、前端指南、部署文档等。

### 3.2 可执行入口区

- `cmd/todoapi/`
  - TODO API 的真实后端入口。
- `cmd/batchrename/`
  - CLI 批量重命名工具入口。
- `cmd/logzip/`
  - 日志归档工具入口。
- `cmd/learn4go/`
  - 最简入口示例。
- `cmd/tinygee-demo/`
  - TinyGee Demo 的入口。

### 3.3 核心业务区

- `internal/todo/`
  - 当前仓库里最完整的后端业务模块。
  - 既包含 HTTP 路由，也包含认证、安全、RBAC、存储抽象。
- `internal/cache/`
  - Redis 示例能力。
- `internal/storage/`
  - MinIO 示例能力。
- `internal/workerpool/`
  - 并发 worker pool 示例。
- `internal/cli/batchrename/`
  - CLI 工具核心逻辑。
- `internal/logzip/`
  - 日志压缩核心逻辑。

### 3.4 前端资源区

- `web/`
  - 本地静态前端页面和前端脚本。
  - 这里没有打包构建链，页面通过 `<script>` 直接引用 JS。

### 3.5 示例与部署区

- `examples/`
  - 各种可运行示例。
  - 其中 `examples/gateway/` 是和前端 / TODO API 直接相关的网关示例。
- `deployments/`
  - Docker、Nginx、Compose 等部署配置。

### 3.6 运行时与脚本

- `start-local.sh`
  - 一键启动 TODO API、Gateway、前端静态服务。
- `stop-local.sh`
  - 停止本地启动的服务。
- `test-smoke.sh`
  - 冒烟测试脚本。
- `logs/`
  - 运行日志目录，属于运行时产物。

## 4. `web/` 前端结构

## 4.1 页面文件

- `portal.html`
  - 学习门户/总入口。
  - 同时承担：
    - 学习章节导航
    - 项目入口导航
    - 服务状态检查
    - 示例命令展示
  - 页面内有较大的内联脚本，负责健康检查、进度状态、示例列表渲染。

- `index.html`
  - 仓库首页，偏静态介绍页。
  - 主要是导航、特性说明、学习路径说明。
  - 基本没有复杂动态逻辑。

- `projects.html`
  - 项目实战页，偏静态卡片展示。
  - 主要展示 CLI、TODO API、Gateway 等项目入口与命令。
  - 交互很轻，主要是按钮提示和跳转。

- `login.html`
  - 旧版登录演示页。
  - 同时兼容：
    - `config.js` 的统一 API 基址
    - `mock-api.js` / `mock-data.js` 的 mock 模式
    - `web/js/utils.js` 的 UI 工具能力
  - 这页更像“兼容旧 RBAC 示例 + 当前 TODO 登录能力”的过渡页面。

- `todo-login.html`
  - 当前最短的真实登录演示页。
  - 用真实 TODO API 的 `/v1/login`、`/v1/refresh`、`/v1/todos` 串起完整链路。
  - 推荐把它作为前后端联调排障的第一入口。

- `admin.html`
  - 管理后台页。
  - 主要演示：
    - 当前用户信息读取
    - 用户列表
    - 角色列表
    - 权限列表
    - 审计日志查看
  - 页面体量较大，存在较多内联脚本逻辑。

- `log-detective.html`
  - 日志分析演示页。
  - 前端形态完整，但后端口径仍偏旧链路，调用的是 `/gateway/log-detective/analyze`。
  - 当前更适合作为“旧演示页 / 预留演示页”看待。

## 4.2 前端脚本文件

- `config.js`
  - 统一决定前端运行环境是 Docker/Nginx 代理模式还是本地开发直连模式。
  - 页面一般先从这里拿 `window.AppConfig`。

- `common.js`
  - 公共 API 客户端、日志面板、部分通用渲染辅助。
  - 主要供 `admin.html`、`log-detective.html` 这类管理/工具页复用。

- `auth-helper.js`
  - Token 存取和自动 refresh 的轻量认证桥接层。
  - 目前主要服务 `todo-login.html`。

- `mock-api.js`
  - fetch 拦截器。
  - 在 `?mock=true` 或 `localStorage.mockApi=true` 时拦截请求并返回 mock 数据。

- `mock-data.js`
  - mock 数据源。
  - 给 `mock-api.js` 提供静态响应样本。

- `js/utils.js`
  - 前端 UI 工具层。
  - 包含 Toast、Loading、表单校验、国际化等辅助能力。

## 5. 后端结构：入口层 / API层 / 安全层 / 数据层

## 5.1 入口层

- `cmd/todoapi/main.go`
  - 后端服务真实入口。
  - 负责：
    - 读取环境变量
    - 选择存储实现（memory / sqlite / mysql）
    - 准备 JWT 配置
    - 创建 `todo.Server`
    - 启动 HTTP 服务
    - 处理优雅关闭

## 5.2 API / 桥接层

- `internal/todo/handler.go`
  - 这是整个 TODO 后端的中心文件。
  - 负责：
    - 路由注册
    - HTTP 请求解析
    - 响应输出
    - 将请求转发到认证、安全、用户存储、TODO 存储等模块

## 5.3 认证与安全层

- `internal/todo/auth.go`
  - JWT 生成、解析、认证中间件。
- `internal/todo/security.go`
  - 邮箱/密码校验、登录失败限制、refresh token 的签发和验证。
- `internal/todo/rbac.go`
  - 角色权限映射、资源/动作解析、授权中间件。
- `internal/todo/validator.go`
  - 请求结构体验证定义。

## 5.4 用户与权限数据层

- `internal/todo/user_store.go`
  - 用户存储抽象。
  - 当前默认实现是内存用户存储，包含预置账号。

## 5.5 TODO 数据层

- `internal/todo/store.go`
  - 内存版 TODO 存储。
  - 适合本地演示和最小运行。
- `internal/todo/store_db.go`
  - GORM 版 TODO 存储。
  - 提供 SQLite / MySQL 实现。

## 5.6 网关层

- `examples/gateway/gin/main.go`
  - Gin 版网关。
- `examples/gateway/stdlib/main.go`
  - Go 标准库版网关。

这两份代码主要做同一件事：

```text
/api/v1/todos...
  → 去掉 /api 前缀
  → 代理到 http://localhost:8080/v1/todos...
```

## 6. 页面与脚本的对应关系

## 6.1 `portal.html`

### 页面职责

- 学习门户首页
- 服务健康状态展示
- 章节导航
- 示例命令 / 示例入口汇总

### 主要脚本关系

- 依赖 `config.js` 提供 `AppConfig`
- 页面内联脚本负责：
  - 判断运行环境
  - 轮询 TODO API / Gateway 健康状态
  - 渲染示例卡片和学习项

### 主要后端关系

- `GET {todoApiBaseUrl}/healthz`
- `GET {gatewayUrl}/health`

## 6.2 `index.html`

### 页面职责

- 项目介绍页 / 落地页
- 主要是静态内容说明

### 脚本关系

- 仅依赖 Bootstrap bundle
- 没有复杂业务逻辑

## 6.3 `projects.html`

### 页面职责

- 展示 CLI、TODO API、Gateway 等项目卡片

### 脚本关系

- 主要是静态展示
- 局部按钮通过 `onclick` 做源码位置提示

## 6.4 `login.html`

### 页面职责

- 旧版登录演示页
- 兼容 mock 和真实 API

### 主要脚本关系

```text
login.html
  → mock-data.js
  → config.js
  → mock-api.js
  → js/utils.js
  → 页面内联登录逻辑
```

### 真实链路

```text
提交登录
  → POST {API_BASE}/v1/login
  → internal/todo.handleLogin
  → auth.go / security.go / user_store.go
```

### 额外说明

- 这页同时带有“旧演示页”和“当前可继续复用的登录页”双重属性。
- 如果 mock 开启，请求会先被 `mock-api.js` 拦截，不会真正打到后端。

## 6.5 `todo-login.html`

### 页面职责

- 演示 Access Token + Refresh Token + 受保护接口调用

### 主要脚本关系

```text
todo-login.html
  → config.js
  → auth-helper.js
  → 页面内联脚本
```

### 真实调用链

```text
点击“登录并获取 Token”
  → POST {todoApiBaseUrl}/v1/login
  → internal/todo.handleLogin
  → issueTokens()
  → 返回 access + refresh
  → AuthClient.setTokens()

页面加载或后续请求
  → AuthClient.authFetch({todoApiBaseUrl}/v1/todos)
  → 如 access 失效且 refresh 存在
      → POST /v1/refresh
      → internal/todo.handleRefresh
      → 重新发起 /v1/todos
```

这是当前仓库里最清楚、最短的一条页面 → JS → API → Service 链路。

## 6.6 `admin.html`

### 页面职责

- 管理后台
- 展示用户、角色、权限、日志等管理面板

### 主要脚本关系

```text
admin.html
  → mock-data.js
  → config.js
  → mock-api.js
  → common.js
  → 页面内联后台逻辑
```

### 真实接口关系

- `GET /v1/me`
- `GET /v1/users`
- `PATCH /v1/users/{id}`
- `GET /v1/rbac/roles`
- `GET /v1/rbac/permissions`

### 额外说明

- 页面内联脚本很重，是当前前端里典型的“页面和脚本写在一个文件里”的后台页。
- 若排障：
  1. 先看页面底部内联脚本
  2. 再看 `common.js`
  3. 再看 `handler.go` 中对应接口

## 6.7 `log-detective.html`

### 页面职责

- 日志输入、分析触发、结果统计和表格展示

### 主要脚本关系

- `config.js`
- `common.js`
- 页面内联脚本负责：
  - 读取输入框文本
  - 调 `apiClient`
  - 渲染统计卡片与结果表格

### 当前接口关系

```text
POST {API_BASE}/gateway/log-detective/analyze
```

### 额外说明

- 这条链路仍带有旧的 Python/日志服务演示口径。
- 当前仓库里没有与它严格对齐的 Go 后端实现，因此更适合作为历史演示页理解，不宜当主链路排障入口。

## 7. 核心调用链拆解

## 7.1 登录链路

```text
页面:
  web/todo-login.html 或 web/login.html

前端:
  fetch(...) / AuthClient.authFetch(...)

后端入口:
  cmd/todoapi/main.go
    → todo.NewServer(...)

API层:
  internal/todo/handler.go
    → handleLogin

认证/安全层:
  internal/todo/user_store.go     查用户
  internal/todo/auth.go           校验密码、签发 JWT
  internal/todo/security.go       登录失败限制、签发 refresh
```

## 7.2 TODO 列表链路

```text
页面:
  web/todo-login.html

前端:
  AuthClient.authFetch(`${todoApiBaseUrl}/v1/todos`)

后端:
  authMiddleware
    → authzMiddleware
    → /v1/todos GET handler
    → user_store.go 查当前用户角色
    → store.go / store_db.go 取 TODO 列表
```

## 7.3 创建 TODO 链路

```text
页面或脚本:
  POST /v1/todos

后端:
  authMiddleware
    → authzMiddleware
    → handler.go 中 /v1/todos POST
    → store.Create(title, userID)
    → store.go 或 store_db.go
```

## 7.4 刷新 Token 链路

```text
前端:
  auth-helper.js / authFetch()
  收到 401 后尝试 refreshToken()

后端:
  handler.go / handleRefresh
    → security.go / validateRefresh
    → security.go / issueTokens
    → 返回新的 access + refresh
```

## 7.5 通过网关访问 TODO API

```text
前端:
  请求 /api/v1/todos

网关:
  examples/gateway/gin/main.go
  或 examples/gateway/stdlib/main.go

代理动作:
  /api/v1/todos
    → 去掉 /api
    → http://localhost:8080/v1/todos

后端:
  internal/todo/handler.go
```

## 8. 排查问题时先看哪里

## 8.1 页面打不开或区域不对

先看：

1. 对应 `web/*.html`
2. 页尾内联脚本
3. 是否依赖 `config.js`

## 8.2 登录失败

先看：

1. `web/todo-login.html` 或 `web/login.html`
2. `web/auth-helper.js`
3. `internal/todo/handler.go` 的 `handleLogin`
4. `internal/todo/security.go`
5. `internal/todo/user_store.go`

## 8.3 TODO 请求返回 401 / 403

先看：

1. `web/auth-helper.js` 是否带上 token
2. `internal/todo/auth.go` 的 `authMiddleware`
3. `internal/todo/rbac.go` 的 `authzMiddleware`

## 8.4 TODO 数据不对

先看：

1. `internal/todo/handler.go` 中 `/v1/todos` 的 GET/POST/PUT/DELETE 分支
2. `internal/todo/store.go`
3. `internal/todo/store_db.go`

## 8.5 网关转发不通

先看：

1. `examples/gateway/gin/main.go`
2. `examples/gateway/stdlib/main.go`
3. `web/config.js`
4. `start-local.sh`

  - 当前真实可跑的 Go 主链路
